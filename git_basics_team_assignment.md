# Практичне завдання: Основи Git та GitHub (командна робота)

## Загальна інформація

* Формат роботи: **команди по 2–3 студенти**

У цьому завданні ви працюєте за спрощеною моделлю, наближеною до реальних командних процесів:

* `main` — стабільна гілка ("production")
* `develop` — основна гілка для розробки
* індивідуальні гілки — короткоживучі гілки для змін окремих розробників

⚠️ **Прямі зміни в `main` заборонені.** Усі зміни потрапляють у `main` тільки після попереднього об’єднання через `develop`.

### Завдання 1. Створення репозиторію

1. Один учасник команди створює новий GitHub-репозиторій.
2. Репозиторій має містити файл `README.md`.
3. Інші учасники команди отримують доступ до репозиторію.

### Завдання 2. Підготовка гілки develop

1. У репозиторії має бути створена окрема гілка `develop`.
2. Гілка `develop` вважається основною гілкою для щоденної розробки.
3. Усі подальші індивідуальні гілки створюються **від `develop`**.

### Завдання 3. Робота у власних гілках

1. Кожен студент працює **у власній гілці**.
2. У репозиторії створіть директорію `students/`.
3. Кожен студент додає **окремий Markdown-файл** зі своїм ім’ям.
4. У файлі має бути:

   * ім’я та прізвище
   * будь-яка інформація про себе

### Завдання 4. Успішний merge в develop

1. Зміни кожного студента **не повинні перетинатися** з файлами інших.
2. Виконайте об’єднання гілок з основною гілкою так, щоб:

   * merge відбувся **без конфліктів**
   * історія змін збереглася коректно

Результат:

* всі індивідуальні файли студентів доступні в основній гілці

### Завдання 5. Створення конфлікту між feature-гілками

1. Два різні студенти:

   * працюють у **двох різних гілках**
   * змінюють **один і той самий фрагмент** файлу `README.md`
2. Обидві гілки мають бути готові до об’єднання з основною.

### Завдання 6. Вирішення конфлікту в develop

1. Під час об’єднання виникає конфлікт.
2. Команда:

   * аналізує причину конфлікту
   * вручну вирішує його
   * зберігає фінальну узгоджену версію файлу

Результат:

* конфлікт вирішений
* основна гілка містить коректний `README.md`

### Завдання 7. Симуляція багу

1. Один з учасників команди навмисно вносить помилку ("баг") у репозиторій.
2. Баг має бути зафіксований у гілці `develop` і помітний для всієї команди.
3. У `README.md` або окремому файлі додайте короткий опис багу:

   * у чому полягає проблема
   * як очікувалось, що система має поводитись

### Завдання 8. Виправлення багу

1. Кожен учасник команди:

   * створює **власний коміт** з виправленням або покращенням, пов’язаним із цим багом
   * працює **в одній спільній гілці для виправлення**
2. У результаті:

   * в гілці має бути **щонайменше один коміт від кожного учасника**
   * історія комітів має чітко відображати внесок кожного

### Завдання 9. Revert або Cherry-pick

Виконайте **один із варіантів (на вибір команди)**:

**Варіант A — Revert**

1. Оберіть коміт, який вніс баг або некоректну зміну.
2. Скасуйте його так, щоб історія репозиторію збереглася.

**Варіант B — Cherry-pick**

1. Оберіть корисний коміт з іншої гілки.
2. Перенесіть його в поточну гілку виправлення.

Результат:

* команда розуміє різницю між виправленням історії та вибірковим перенесенням змін

### Завдання 10. Pull Request з develop у main

1. Після завершення всіх змін та вирішення конфліктів команда вважає гілку `develop` готовою до релізу.
2. Створюється **Pull Request з `develop` у `main`**.
3. Pull Request має містити:

   * опис виконаних задач
   * короткий підсумок змін (release notes)
4. Перед об’єднанням Pull Request:

   * усі учасники команди переглядають зміни
   * залишають коментарі або підтверджують готовність до merge
5. Pull Request має бути успішно об’єднаний.


## Приклади назв гілок (обов’язкові до використання)

Під час виконання завдань команда має дотримуватись єдиного підходу до іменування гілок.

### Основні гілки

* `main` — стабільна версія проєкту
* `develop` — основна гілка для розробки

### Індивідуальні гілки студентів (feature)

Кожен студент працює у власній гілці, створеній від `develop`.

Приклади:

* `feature/student-oleksii`
* `feature/student-maria`
* `feature/student-andrii`

### Гілки для змін, що не викликають конфліктів

Використовуються для додавання нових файлів або незалежних змін.

Приклади:

* `feature/add-student-profile`
* `feature/update-team-info`

### Гілки для навмисного створення конфлікту

Дві різні гілки змінюють один і той самий фрагмент файлу.

Приклади:

* `feature/readme-edit-oleksii`
* `feature/readme-edit-maria`

### Гілка для спільного виправлення багу

У цій гілці працюють **усі учасники команди**, кожен робить щонайменше один коміт.

Приклади:

* `bugfix/shared-readme-bug`
* `bugfix/team-bugfix`
